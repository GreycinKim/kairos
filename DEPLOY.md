# Deploy Kairos (Vercel + Render)

Monorepo layout: **`frontend/`** (Vite SPA), **`backend/`** (FastAPI + Postgres).

## 1. Render — API and database

1. Push this repo to GitHub (or GitLab / Bitbucket Render supports).
2. In [Render](https://dashboard.render.com) → **New** → **Blueprint**.
3. Connect the repo and select **`render.yaml`** at the repo root.
4. When prompted, set **`CORS_ORIGINS`** to your real frontend origins, comma-separated, for example:
   - `https://your-app.vercel.app`
   - After you add a custom domain, include that too, e.g. `https://app.yourdomain.com`
5. Finish the deploy and copy the web service URL, e.g. `https://kairos-api-xxxx.onrender.com`.

**Notes**

- **Python version:** Render defaults new services to **3.14**. This repo pins **3.12** (`render.yaml` → `PYTHON_VERSION`, repo-root `.python-version`) so packages like `pydantic-core` install from **wheels** instead of compiling with Rust.
- The first request after idle can be slow on the **Starter** web plan (cold start).
- `DATABASE_URL` from Render is normalized in `backend/app/config.py` for **asyncpg**; if `SYNC_DATABASE_URL` still pointed at localhost, it is derived from `DATABASE_URL` for Alembic.
- Migrations run at **process start** (`alembic upgrade head && uvicorn …`) because Render **free** web services do not support `preDeployCommand`. On a paid tier you can switch to `preDeployCommand` for migrations-only-on-deploy.
- `JWT_SECRET` is generated by the blueprint; you can rotate it in the service **Environment** tab.

**Health check:** `GET https://<your-service>.onrender.com/api/health` → `{"status":"ok"}`.

## 2. Vercel — frontend

1. In [Vercel](https://vercel.com) → **Add New** → **Project** → import the same repo.
2. Set **Root Directory** to **`frontend`** (important).
3. Framework preset **Vite** is fine; `frontend/vercel.json` sets SPA rewrites and build output **`dist`**.
4. Under **Environment Variables** (Production, and Preview if you want):
   - **`VITE_API_BASE_URL`** = `https://<your-render-service>.onrender.com/api` (no trailing slash; include `/api`).
   - Optional: **`VITE_LOGIN_PASSWORD`**, **`VITE_AUTH_DISABLED`** — see `frontend/.env.production.example` and `src/lib/authSession.ts`.
5. Deploy. Open the Vercel URL and confirm the reader loads and API calls succeed (browser **Network** tab).

**CORS**

- Explicit origins: set **`CORS_ORIGINS`** on Render to your Vercel production URL.
- Any **`https://*.vercel.app`** origin is also allowed via regex in `backend/app/main.py` for previews.

## 3. Smoke test

- [ ] `/api/health` on Render returns OK.
- [ ] Vercel site loads; timeline / notes hit the API without CORS errors.
- [ ] Login screen (if enabled) works.

## Plans and costs

In `render.yaml`, the **web** service uses Render’s **`free`** plan (cold starts after idle). The **Postgres** instance omits `plan` so Render uses the default (**`basic-256mb`** tier). Edit `render.yaml` if you want paid tiers for uptime or capacity.

Vercel Hobby tier is enough for many static + serverless frontends; check current limits on their site.
